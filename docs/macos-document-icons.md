# macOS Native Document Icons — Research & Implementation

## Problem

Netscope declares `.har` as a custom document type, but macOS was showing the pre-rendered `har-icon.icns` file as the full icon instead of generating the native Big Sur+ document shape (white page with folded corner, app icon badge, extension label).

## Root Cause

**`UTTypeIconFile` causes macOS to use the provided `.icns` as the entire icon.** When any icon file is specified — either `UTTypeIconFile` in `UTExportedTypeDeclarations` or `CFBundleTypeIconFile` in `CFBundleDocumentTypes` — macOS renders that file directly. The auto-generated page curl only happens when the UTI declaration has **no icon reference at all**.

Netscope had two icon references fighting for control:
1. `UTTypeIconFile: "har-icon.icns"` in the UTI declaration (injected by `afterPack.js`)
2. `CFBundleTypeIconFile: "har-icon"` in the document type (injected by electron-builder from `fileAssociations.icon`)

Both pointed to a pre-rendered `.icns` that already contained a hand-drawn page curl. macOS displayed it as-is.

## What Was Tried (and why it didn't work)

| Attempt | Result | Why |
|---|---|---|
| `CFBundleTypeIconFile` pointing at our .icns | Icon shown as-is | Custom icon overrides auto-generation |
| `UTExportedTypeDeclarations` with `UTTypeIconFile` | Icon shown as-is | Same — any icon file suppresses auto-generation |
| Remove all icon refs, declare UTI with `public.data` | Shows raw app icon | This was reported but actually *should* work — likely a stale icon cache issue |
| Add `public.json` conformance | JSON text preview | macOS recognizes JSON and renders a QuickLook preview instead |

## Verified Solutions (tested with native Swift app bundles)

Four configurations were tested using minimal `.app` bundles registered with the Netscope app icon:

### Test 1: No icon on UTI (the fix we shipped)
```xml
<key>UTExportedTypeDeclarations</key>
<array><dict>
    <key>UTTypeIdentifier</key><string>com.netscope.har</string>
    <key>UTTypeConformsTo</key><array><string>public.data</string></array>
    <key>UTTypeTagSpecification</key><dict>
        <key>public.filename-extension</key><array><string>har</string></array>
    </dict>
</dict></array>
```
**Result:** Native page curl with Netscope icon badge and "HAR" extension label. Exactly what we wanted.

### Test 2: UTTypeIcons with UTTypeIconText
```xml
<key>UTTypeIcons</key><dict>
    <key>UTTypeIconText</key><string>HAR</string>
</dict>
```
**Result:** Native page curl with Netscope icon badge and custom "HAR" text. Same visual output when the text matches the extension. This is what Pages/Numbers/Keynote use.

### Test 3: UTTypeIconFile (the old approach)
```xml
<key>UTTypeIconFile</key><string>har-icon.icns</string>
```
**Result:** Pre-rendered `.icns` displayed as-is. No auto-generation.

### Test 4: UTTypeIconName without Asset Catalog
```xml
<key>UTTypeIconName</key><string>HARDocument</string>
```
**Result:** Blank icon. The named image lookup fails because Electron apps don't include an Asset Catalog (`Assets.car`).

## How Other Apps Handle This

### Apps that get the native page curl
- **Mail.app** (.eml): No icon on exported UTI at all. Conforms to `public.data, public.email-message`.
- **Pages/Numbers/Keynote** (.pages, .numbers, .key): Use `UTTypeIcons: { UTTypeIconText: "PAGES" }`. Conforms to `com.apple.package, public.composite-content`.
- **Preview.app** (.radiance, .pbm, etc.): No icon on exported UTIs.
- **Xcode** (.swift): Uses `UTTypeIcons: { UTTypeIconText: "swift", UTTypeIconBadgeName: "swift-badge" }` — badge references an Asset Catalog image.

### Apps that provide custom pre-rendered icons
- **Charles Proxy** (.har, .chls, .pcap): Pre-renders the page curl directly into `.icns` files (`charles_har.icns`). No `UTExportedTypeDeclarations` at all — only legacy `CFBundleDocumentTypes` with `CFBundleTypeIconFile`. The page curl you see is artwork baked into the icon, not generated by macOS.
- **VS Code** (.ts, .py, .html, etc.): ~20 custom `.icns` files with pre-rendered icons. No UTI declarations.

### Key insight from Charles
Charles's .har files *look* like they have native page curl treatment, but they don't. Charles hand-drew the page curl shape into its `.icns` files. This is the pre-Big Sur approach. The native system-generated icon has slightly different proportions, shadows, and the app icon badge uses the modern rounded-rectangle style.

## The Fix

Two files changed:

### `scripts/afterPack.js`
- Removed the `UTTypeIconFile` line from UTI injection
- Added a step to delete `CFBundleTypeIconFile` from `CFBundleDocumentTypes:0` (electron-builder injects this from `fileAssociations.icon`)
- Cleaned up the PlistBuddy invocation pattern

### `package.json`
- Removed `icon: "build/har-icon"` from `mac.fileAssociations`
- Removed `extraResources` entry for `har-icon.icns` (no longer needed in the bundle)

## Notes

- The `build/har-icon.icns` file is still in the repo. It could be deleted, but keeping it doesn't hurt and serves as a fallback reference if we ever need to go back to a custom icon.
- `images/netscope-file.png` (the source art for `har-icon.icns`) is also still in the repo.
- The `conformsTo: public.data` chain is correct. Don't add `public.json` — it causes macOS to show JSON text previews. Don't add `public.content` — it's not needed for the page curl and may have side effects with AirDrop/Handoff.
- The native icon generation works on unsigned/ad-hoc-signed builds. No code signing requirements.
- This approach works regardless of whether the app has an Asset Catalog. Electron's loose `.icns` app icon is picked up correctly for the badge.
